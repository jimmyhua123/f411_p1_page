# 2026-05-08 STM32F411 Head-Tracking Audio Report

## 1. Project Summary

This project is an STM32F411CEU6 head-tracking audio prototype. The firmware
reads an MPU6500 IMU over I2C, estimates roll and pitch, classifies head
direction, and drives two MAX98357A I2S amplifier modules for stereo audio.

Current project focus:

- Head-pose controlled stereo audio.
- Button-driven user modes.
- Fault handling and diagnostic output.
- Power-policy demo with measured audio-rail current.
- UART telemetry for external visualization.
- MicroSD CSV logging for later analysis.

One additional related project is available outside this firmware project:

- A separate PC/tool project has been built to read `tel` / `log` data.
- This firmware now produces UART telemetry and CSV log formats that can be
  consumed by that external reader.
- The next integration target is to make the external tool prefer `SIM3,` and
  parse `PWR,` / `[DEMO_SUMMARY]` lines when available.

## 2. Current Completed Results

### 2.1 Sensor and Head Tracking

- MPU6500 I2C bring-up completed.
- MPU6500 WHO_AM_I read verified.
- Accel-only roll and pitch calculation implemented.
- Low-pass attitude filter implemented.
- Head state machine implemented:
  - `CENTER`
  - `LEFT`
  - `RIGHT`
  - `DOWN`
  - `UP`
- Head-state hysteresis implemented to avoid unstable switching.

Key thresholds:

```text
LEFT  enter roll < -25 deg, exit roll > -18 deg
RIGHT enter roll >  25 deg, exit roll <  18 deg
DOWN  enter pitch < -35 deg, exit pitch > -25 deg
UP    enter pitch >  35 deg, exit pitch <  25 deg
```

### 2.2 Audio Output

- Dual I2S DMA audio output completed.
- Left MAX98357A uses I2S2.
- Right MAX98357A uses I2S3.
- 440 Hz sine tone output implemented.
- Roll-based stereo panning implemented.
- `HEAD_DOWN` software mute implemented.
- Volume smoother implemented to reduce click/pop.
- Bench-test user-facing 100% volume is capped to 50% actual gain.
- 20% volume mode remains 20% actual gain.

Audio behavior:

```text
roll =   0 deg -> L=1.00 R=1.00
roll = -30 deg -> L=1.00 R=0.30
roll = +30 deg -> L=0.30 R=1.00

HEAD_DOWN -> L=0.00 R=0.00
MUTED     -> L=0.00 R=0.00
LOW_POWER -> audio rail off
FAULT     -> L=0.00 R=0.00
```

### 2.3 User Interaction and Modes

- PA0 short press toggles master volume between 20% and 100%.
- PA0 long press 1 to 3 seconds toggles `ACTIVE` / `MUTED`.
- PA0 long press 3 seconds or more enters `DIAGNOSTIC`.
- System mode state machine implemented:
  - `ACTIVE`
  - `MUTED`
  - `LOW_POWER`
  - `DIAGNOSTIC`
  - `FAULT`

Mode behavior:

```text
ACTIVE:     IMU 50 Hz, audio on, telemetry 20 Hz
MUTED:      IMU 50 Hz, audio software-muted, telemetry 10 Hz
LOW_POWER:  IMU 10 Hz, audio rail off through AO3401, telemetry 2 Hz
DIAGNOSTIC: left beep, right beep, stereo beep, health logs
FAULT:      audio target forced to zero, I2S DMA keeps running silence
```

Status LED behavior:

```text
BOOT:       PC13 red + PC14 green + PC15 yellow briefly on
ACTIVE:     PC14 green on
MUTED:      PC15 yellow on
LOW_POWER:  PC15 yellow blink
DIAGNOSTIC: PC13 + PC14 + PC15 all on
FAULT:      PC13 red on
```

### 2.4 Fault Handling

- `fault_manager.c` / `fault_manager.h` implemented.
- Fault flags implemented:
  - `IMU_LOST`
  - `I2S_LEFT`
  - `I2S_RIGHT`
  - `AUDIO_CLIP`
  - `BUTTON_STUCK`
  - `SD_LOG`
- `FAULT` mode forces audio target to zero.
- I2S DMA keeps running with silence in fault mode.
- UART fault logs are printed.
- Long press can request fault recovery.
- SD logging fault is non-fatal, so the main audio demo can continue even if SD
  is unavailable.

### 2.5 Power Policy and Audio Rail Cutoff

- `power_policy.c` / `power_policy.h` implemented.
- Mode-based IMU rate, telemetry rate, and audio behavior implemented.
- `MUTED` for 5 seconds automatically enters `LOW_POWER`.
- `LOW_POWER` wake by PA0 button implemented.
- `LOW_POWER` wake by about 10 deg roll/pitch movement implemented.

Policy table:

```text
+------------+----------+-------------+----------------+
| Mode       | IMU Rate | Telemetry   | Audio Policy   |
+------------+----------+-------------+----------------+
| ACTIVE     | 50 Hz    | 20 Hz       | rail on        |
| MUTED      | 50 Hz    | 10 Hz       | software mute  |
| LOW_POWER  | 10 Hz    | 2 Hz        | rail off       |
| DIAGNOSTIC | 50 Hz    | 20 Hz       | rail on        |
| FAULT      | 10 Hz    | 2 Hz        | forced silence |
+------------+----------+-------------+----------------+
```

### 2.6 INA219 / GY-219 Power Measurement

- GY-219 / INA219 wired on shared I2C1 bus.
- INA219 address `0x40` detected.
- 5V audio rail measured through `VIN+ -> VIN-`.
- INA219 sample period: 100 ms.
- 16-sample moving average implemented.
- Per-mode current min/max reset implemented.
- `PWR,` telemetry line implemented.
- `power_report.c` / `power_report.h` per-state summary implemented.
- `[DEMO_SUMMARY]` UART output implemented.

Measured / recorded values:

```text
Before AO3401 rail cutoff:
ACTIVE 20% audio output    ~= 20 mA
HEAD_DOWN software silence ~= 6 mA
MUTED software mute        ~= 6 mA
LOW_POWER v1 software mute ~= 6 mA

After AO3401 v2 rail cutoff:
ACTIVE 20% audio output    ~= 18.0 mA
HEAD_DOWN software silence ~= 5.6 mA
LOW_POWER AO3401 off       ~= -0.3 mA
```

Acceptance result:

```text
AUDIO_RAIL_CUTOFF=PASS

Software silence reduces audio rail current from about 18 mA to about 5.6 mA.
Hardware rail cutoff reduces audio rail current from about 5.6 mA to near 0 mA.
The small negative LOW_POWER value is treated as INA219 low-current offset.
```

Verified INA219 sample logs:

```text

[INA219] bus=4972mV shunt=1010uV current=8.50mA power=50mW
[INA219] bus=4972mV shunt=400uV current=4.00mA power=20mW
[INA219] bus=4972mV shunt=-30uV current=0.00mA power=0mW
```

### 2.7 AO3401 Audio Rail Switch

- AO3401 added after INA219 for switched MAX98357A `5V_AUDIO` rail.
- HW-221 level shifter translates PB8 from 3.3 V control to 5 V gate drive.
- `LOW_POWER` turns the audio rail off.
- Other modes keep the audio rail on for normal operation or debug.

Wiring summary:

```text
5V_AUDIO_IN -> INA219 VIN+
INA219 VIN- -> AO3401 Source
AO3401 Drain -> 5V_AUDIO_SW -> left/right MAX98357A VIN

PB8 -> HW-221 LV input
HW-221 HV output -> AO3401 Gate

PB8 low  -> gate low  -> audio rail on
PB8 high -> gate high -> audio rail off
```

### 2.9 UART Telemetry

- Legacy `SIM,` telemetry implemented.
- `SIM2,` mode + fault telemetry implemented.
- `SIM3,` power-policy telemetry implemented.
- `PWR,` power monitor telemetry implemented.
- `[PWR]` power summary lines implemented.
- `[DEMO_SUMMARY]` diagnostic report implemented.

Current telemetry formats:

```text
SIM,<ms>,<roll_cd>,<pitch_cd>,<head>,<target_l_pct>,<target_r_pct>,<smooth_l_pct>,<smooth_r_pct>

SIM2,<ms>,<mode>,<fault_hex>,<roll_cd>,<pitch_cd>,<head>,<target_l_pct>,<target_r_pct>,<smooth_l_pct>,<smooth_r_pct>

SIM3,<ms>,<mode>,<fault_hex>,<imu_hz>,<tel_hz>,<audio_en>,<roll_cd>,<pitch_cd>,<head>,<target_l_pct>,<target_r_pct>,<smooth_l_pct>,<smooth_r_pct>

PWR,mode=<mode>,bus=<bus_v>,current=<current_ma>,power=<power_mw>,avg=<avg_ma>,min=<min_ma>,max=<max_ma>,n=<sample_count>
```

Example:

```text
SIM3,1234,ACTIVE,00000000,50,20,1,-1850,420,CENTER,20,20,20,20
PWR,mode=ACTIVE,bus=4.972,current=17.20,power=86.0,avg=16.80,min=15.90,max=18.30,n=16
```

## 7. Short Report Conclusion

The project has evolved from a basic head-tracking audio prototype into a
product-like embedded demo. It now includes sensor input, attitude filtering,
head-state classification, stereo audio control, button interaction, diagnostic
mode, fault handling, power-policy behavior, measured audio-rail current, UART
telemetry, and MicroSD CSV logging.

The most important verified result is the LOW_POWER audio-rail cutoff. Software
mute lowers the audio rail from about 18 mA to about 5.6 mA, and the AO3401
hardware cutoff lowers it to near 0 mA. The project also has structured data
outputs, so the separate tel/log reader project can be used as the next step
for visualization, analysis, and demo reporting.
